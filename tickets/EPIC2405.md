# EPIC2405 — `payment_accounts` config + BE auto-resolve `revenueAccountId` + FE rewire

> Exported plan (2026-05-24). Branch: `EPIC-18052026-cash-vouchers`. Status: **planned, not yet implemented.**

## Context

POS checkout (`POST /invoices/:id/checkout`) currently trusts the frontend to send both
`revenueAccountId` and each payment line's `accountId`. This caused a production-style bug:
the FE mapper hardcodes a junk id (`invoicePayloadMapper.ts:131` →
`line.cashAccountId = "f29226ef..."`) with its validation commented out
(`invoicePayloadMapper.ts:139-147`). That id exists in neither `accounts` nor `cash_accounts`,
so checkout returns 200 but the async journal consumer throws
`Account f29226ef... not found` and the event lands in the DLQ — a silent failure.

Two root problems:
1. **Payment account is unconstrained.** A bank/card payment can land in many different
   accounts (ACB, MB, VCB...). The cashier must pick *which* receiving account, but today the
   FE filters `/accounts?type=ASSET` (too broad — includes TK 1311 receivable) and there is no
   structured "Tài khoản thu" config. Reference UI: a payment-method row + a "Tài khoản thu"
   dropdown showing `4242424242424242424 - ABB - Ngân hàng TMCP An Bình`.
2. **`revenueAccountId` is FE knowledge it shouldn't own.** Revenue account (TK 511) is an
   accounting concept; the cashier shouldn't choose it. It should be resolved server-side.

### Decisions (confirmed with user)
- **Model:** one `payment_accounts` table with structured bank columns (`bank_name`,
  `bank_code`, `account_number`) mapping to a COA account. Keep the API enum
  `cash | bank_transfer | card` unchanged (names are display-only; accounting logic untouched).
- **Default revenue:** generic role-based `accounting_default_account` table (extensible to
  receivable later), resolved server-side branch→org→throw.
- **Admin UI:** API + seed only this round (backoffice management page = follow-up).
- **Validation:** strict whitelist — each payment line `accountId` must be a configured
  `payment_accounts` row for that branch+method, rejected synchronously (400) at checkout.
- **Scope:** only `checkout` (the `/debt` route shares the same DTO; FE only hits `/checkout`).
  `exchange.service.ts` / `return.service.ts` keep sending `revenueAccountId` — deferred
  follow-up (the resolver is designed generic so they can adopt it later).
- **Out of scope:** legacy `POST /pos/sales/checkout` (`CheckoutService`/`CheckoutDto`) — not
  called by pos-web.

## Backend

### 1. Migration
New file `apps/api/src/database/migrations/1781300000000-PaymentAccountsAndDefaultAccounts.ts`
(timestamp strictly greater than the latest `1781200000000`). Hand-written raw SQL via
`queryRunner.query()`, style per `1781000000000-CashVouchersPhase1.ts` (uuid PK
`DEFAULT uuid_generate_v4()`, `varchar` org/branch ids, `TIMESTAMP` audit cols, `PK_`/`UQ_`/`FK_`
naming, partial unique indexes `WHERE deleted_at IS NULL`).

```
CREATE TYPE payment_account_method_enum AS ENUM ('cash','bank_transfer','card');
CREATE TYPE accounting_default_account_role_enum AS ENUM ('REVENUE','RECEIVABLE'); -- ship both up front

CREATE TABLE payment_accounts (
  id uuid PK default uuid_generate_v4(),
  organization_id varchar NOT NULL,
  branch_id       varchar NOT NULL,                    -- NOT NULL (diverge from BaseEntity)
  payment_method  payment_account_method_enum NOT NULL,
  account_id      uuid NOT NULL,                        -- FK -> accounts(id) ON DELETE RESTRICT
  bank_name       varchar(200) NULL,                   -- NULL for cash
  bank_code       varchar(50)  NULL,
  account_number  varchar(50)  NULL,
  label           varchar(255) NULL,                   -- e.g. "Tiền mặt" / optional channel name
  is_active       boolean NOT NULL DEFAULT true,
  sort_order      int NOT NULL DEFAULT 0,
  created_at/updated_at/deleted_at TIMESTAMP, created_by varchar NOT NULL,
  -- partial unique: one mapping per (org,branch,method,account) among live rows
  UNIQUE INDEX (organization_id,branch_id,payment_method,account_id) WHERE deleted_at IS NULL,
  INDEX (organization_id,branch_id), INDEX (account_id),
  FK account_id -> accounts(id) ON DELETE RESTRICT
);

CREATE TABLE accounting_default_account (
  id uuid PK, organization_id varchar NOT NULL,
  branch_id varchar NULL,                              -- NULL = org default; set = branch override
  account_role accounting_default_account_role_enum NOT NULL,
  account_id uuid NOT NULL,                            -- FK -> accounts(id) ON DELETE RESTRICT
  created_at/updated_at/deleted_at TIMESTAMP, created_by varchar NOT NULL,
  -- Postgres treats NULLs as distinct, so use TWO partial unique indexes:
  UNIQUE INDEX (organization_id,account_role)            WHERE branch_id IS NULL     AND deleted_at IS NULL,
  UNIQUE INDEX (organization_id,branch_id,account_role)  WHERE branch_id IS NOT NULL AND deleted_at IS NULL
);
```
`down()` drops tables then types.

### 2. New module `apps/api/src/modules/accounting/payment-accounts/`
Hand-built (mirrors `accounting/cash/` — the closest analog: branch-scoped tender config linked
to COA via `account_id`). Files:

- `payment-account.entity.ts` — extends `BaseEntity`, overrides `branch_id` to `nullable:false`,
  `@DeleteDateColumn`, the enum + bank columns above.
- `accounting-default-account.entity.ts` — extends `BaseEntity` (branchId nullable kept).
- `enums.ts` — `PaymentAccountMethod` (string values mirror `InvoicePaymentMethod`:
  `cash`/`bank_transfer`/`card`) and `AccountingDefaultAccountRole` (`REVENUE`/`RECEIVABLE`).
- `payment-account.service.ts`:
  - `listForPos(organizationId, branchId, method?)` → flat `PaymentAccountEntity[]`, active only,
    `ORDER BY sort_order, label` (createQueryBuilder, org + explicit branchId + optional method —
    do **not** use BaseCrudService branch scoping, which would force `actor.branchId`).
  - `create/update/softDelete(...)` for admin (mirror `CashService.createAccount`).
  - `validatePaymentLines(lines:{accountId,paymentMethod}[], organizationId, branchId)` — **one**
    query fetching active `payment_accounts` JOIN `accounts` (both `is_active`) for the branch,
    build `Set<"method:accountId">`, throw `BadRequestException` for any line not in the set.
- `default-account-resolver.service.ts`:
  - `resolveRevenueAccountId(organizationId, branchId?)` → calls generic
    `resolve(role, organizationId, branchId?)`.
  - Single query: `WHERE org AND role AND (branch_id=$b OR branch_id IS NULL) AND deleted_at IS NULL
    ORDER BY branch_id NULLS LAST LIMIT 1` (branch override wins, falls back to org default).
  - No row → `BadRequestException("No revenue account configured for branch <id>. Configure a
    default revenue account before checkout.")` — **never** fall back to a hardcoded account.
- `payment-account.controller.ts` — `@Controller('payment-accounts')`, guards/interceptor exactly
  like `cash.controller.ts` (`AuditInterceptor`, `PermissionGuard`, `BranchScopeGuard`,
  `@RequireBranchScope()`):
  - `GET /payment-accounts?branchId=&method=` → `listForPos` (perm `accounting.payment_account.read`).
  - `POST` / `PATCH /:id` / `DELETE /:id` (perms `accounting.payment_account.{create,update,delete}`).
  - `GET /payment-accounts/default-accounts?role=&branchId=` + `PUT /payment-accounts/default-accounts`
    (upsert) for the default-account config (perms `accounting.default_account.{read,manage}`).
- `dto/` — `create-payment-account.dto.ts`, `update-payment-account.dto.ts`,
  `upsert-default-account.dto.ts` (class-validator + `@ApiProperty`).
- `payment-accounts.module.ts` — `TypeOrmModule.forFeature([PaymentAccountEntity,
  AccountingDefaultAccountEntity])`, declares controller + both services, **exports**
  `PaymentAccountService` + `DefaultAccountResolverService`.

Wire into `apps/api/src/modules/accounting/accounting.module.ts` imports **and** exports (mirror
`CashModule`). `pos.module.ts` already imports `AccountingModule` (line 64) — no pos wiring needed.

## Sequence — resolver fallback
```
resolveRevenueAccountId(org, branch)
  → SELECT ... WHERE org AND role=REVENUE AND (branch_id=branch OR branch_id IS NULL)
       ORDER BY branch_id NULLS LAST LIMIT 1
  → branch override row?  yes → use it
                          no  → org default row? yes → use it
                                                 no  → throw 400 (configure revenue account)
```
